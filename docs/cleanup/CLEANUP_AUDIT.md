# Cleanup Audit

This audit classifies cleanup opportunities by risk. It is based on repository evidence from source files, tests, and current documentation. It does not recommend broad rewrites because this repository is a scientific LiDAR pipeline where reproducibility, coordinate conventions, calibration behavior, filtering, and output schemas are behavior-sensitive.

## Findings

| Issue | File / Function | Evidence | Why it matters | Proposed fix | Risk level | Recommendation |
| --- | --- | --- | --- | --- | --- | --- |
| Stray no-op expression in config defaults | `lidar_analysis/config.py::default_analysis_yaml_dict` | The line `d# Backward compatibility:` is parsed as a no-op expression `d` followed by a comment. | It is confusing and looks accidental. | Replace it with a normal comment. | Safe cleanup | Change now. |
| Fusion output column contract is implicit | `pipeline_core.choose_fusion_method`, `fusion.fuse_by_time`, `fusion_imu_interp.fuse_by_imu_interp`, `fusion_pps.fuse_by_pps` | Downstream code indexes fused arrays as columns 0 through 8 for time, angles, distance, RSSI, encoder, and IMU fields. | Magic column indexes make future fusion changes risky. | Add comments documenting the fused row contract. Longer-term, consider named constants or a small schema helper. | Safe cleanup for comments; behavior-sensitive for structural change | Add comments now; defer structural changes. |
| Coordinate sign convention is critical but easy to miss | `pipeline_core._to_cartesian_mm`, `pipeline_core.reconstruct_world_points` | `_to_cartesian_mm` returns `[x, -y, -z]`; docs define `X` left/right, `Y` vertical, `Z` travel direction. | A small sign change would alter all point clouds and traits. | Add a concise warning comment near conversion/reconstruction. | Safe cleanup for comments; risky for math changes | Add comments now; do not change math. |
| Internal units alternate between millimeters and meters | `pipeline_core.reconstruct_world_points`, `pipeline_core.Plot.write`, `pointcloud_ops._voxel_count`, `pointcloud_ops._bilateral_scalar_filter` | Pipeline calculations often use millimeters; CSV output uses meters; point-cloud operation config values often use meters. | Unit mistakes can silently change filtering and trait values. | Add focused comments where values cross the mm/m boundary; keep docs aligned. | Safe cleanup for comments; behavior-sensitive for unit conversion changes | Add comments now; avoid conversion changes. |
| Legacy directory runner appears stale | `pipeline_core.run_experiment` and `pipeline_core.run_for_directory` | Active path is `central_runner.run_experiment_date` -> `pipeline_stages.ProcessScanStage`. `pipeline_core.run_experiment` references undefined `d` when calling `run_for_directory`. | This can confuse maintainers and may be broken if called directly. | Mark as legacy in comments and plan a separate fix or test if it must remain supported. | Behavior-sensitive cleanup | Document now; do not change behavior without approval. |
| Legacy calibration loader exits the process | `pipeline_core.load_calibration` | Uses `sys.exit(1)` for validation errors. Active runner uses `central_runner.read_calibration_from_cart_config`. | Process exits are hard to test and inconsistent with active runner exceptions. | Leave behavior unchanged; consider replacing with exceptions only if legacy path is still supported and tests are added. | Behavior-sensitive cleanup | Document only. |
| Open3D/topology compatibility fields remain in config | `config.AnalysisConfig`, `pipeline_core.analyze_plot` | `write_o3d_ply`, `run_o3d_metrics`, and `run_topology` are labeled compatibility shims. `Plot._write_ply` returns immediately. | Maintainers may assume these options are active recommended paths. | Keep fields for compatibility; document that active topology should use `pointcloud_ops: [{op: topology_trait}]`. | Behavior-sensitive cleanup | Document now; removal requires approval. |
| `write_lidar_per_plot` appears unused in active path | `config.AnalysisConfig`, `central_runner.build_config` | Field is parsed, but source search found no active use in `pipeline_core.py` beyond config wiring. | Dead or ambiguous config keys create user confusion. | Keep for compatibility; document as ambiguous until a test or owner decision confirms removal. | Behavior-sensitive cleanup | Document only. |
| Duplicate smoke tests | `tests/test_mark_splitting_smoke.py`, `tests/test_splitting_style_smoke.py` | `cmp -s` returns identical files. | Duplicate tests increase maintenance cost and may diverge later. | Keep one canonical file and remove the duplicate in a test-cleanup batch, or convert one into a pytest import wrapper. | Safe cleanup if approved; test-suite change | Plan for Batch 3, not now. |
| Tests are partly script-style | Multiple `tests/test_*_smoke.py` files | Several tests define `main()` and print `PASS`; some are not collected as pytest tests unless they expose `test_*` functions. | Test expectations are harder to run consistently. | Convert script-style tests to pytest functions in a dedicated batch. | Safe cleanup if behavior preserved | Plan for Batch 3. |
| YAML loading is inconsistent | `central_runner.py`, `pipeline_core.py`, `orchestrator.py`, `run_manifest.py`, `yaml_loader.py` | Several modules import `yaml` directly; `yaml_loader.py` provides a fallback loader but is not used by active runner/core modules. | Dependency errors are inconsistent and docs must explain PyYAML is required. | Either standardize on direct PyYAML with a dependency manifest or route all modules through `yaml_loader.py`. | Behavior-sensitive cleanup | Document now; implementation requires approval and tests. |
| README is stale relative to docs and code | `README.md` | Lists `topology.py`, while topology code lives under `lidar_analysis/topology/`. The Important Files code block is not visibly closed in the inspected output. | Public entry-point documentation can mislead readers. | Update README to point to `docs/` and correct stale file references. | Safe cleanup | Good candidate for documentation-only batch. |
| Debug logging is verbose in core processing | `pipeline_core.apply_global_filters`, `pipeline_core.process_scan`, fusion modules | Multiple unconditional `print` statements emit debug ranges and per-scan details. | Logs may be noisy for batch runs; changing logs can affect troubleshooting expectations. | Introduce a configurable verbosity/logging strategy only in a behavior-sensitive batch. | Behavior-sensitive cleanup | Document only. |
| `pipeline_core.py` is large | `lidar_analysis/pipeline_core.py` | File is 1887 lines and includes legacy calibration, reconstruction, filtering, splitting, traits, and directory running. | Large files increase change risk and make reviews harder. | Extract only narrow helpers after tests cover behavior; avoid broad rewrites. | Behavior-sensitive cleanup | Plan later, not now. |
| Result and point-cloud schemas are spread across code paths | `central_runner.phenotype_columns`, `pipeline_core.Plot.write`, `pipeline_core.run_for_directory` | Active runner writes `results.csv`; legacy directory runner builds a separate traits table shape. | Schema drift can break downstream analysis. | Document active schemas; consider schema constants only with tests. | Risky / do not change without approval | Document only. |
| Coordinate transformation and IMU correction are high-risk | `pipeline_core._to_cartesian_mm`, `pipeline_core.reconstruct_world_points`, `fusion_imu_interp.py` | Coordinate signs, rotation order `R.from_euler("ZXY", ...)`, and IMU zeroing directly affect geometry. | Any change changes scientific output. | Add comments only; require fixture-based validation before math changes. | Risky / do not change without approval | Comments only. |
| RSSI normalization math is behavior-sensitive | `pipeline_core.apply_rssi_normalization_after_masks`, `normalize_rssi_by_phi_percentile`, `normalize_rssi_by_phi_zscore` | Normalization is recomputed after global masks and preserves raw RSSI in column 3. | Changing order or columns changes scalar fields and filtering results. | Document contract; do not alter math. | Risky / do not change without approval | Comments only. |
| LAI/FAD/topology trait formulas should be treated as scientific code | `lidar_analysis/lai/fad.py`, `lidar_analysis/lai/lai.py`, `lidar_analysis/topology/stand_count.py` | These modules implement trait formulas and legacy conversions. | Formula changes affect scientific interpretation. | Only add clarifying comments unless there is a tested bug fix. | Risky / do not change without approval | Document only. |

## Recommended Immediate Scope

Change now:

- Clean the stray no-op `d` expression in `config.default_analysis_yaml_dict`.
- Add concise comments for fused row columns, coordinate conventions, and units at high-risk boundaries.
- Add a cleanup changelog for all uncommitted changes.

Document only for now:

- Duplicate smoke tests.
- Legacy `pipeline_core.run_experiment` / `run_for_directory`.
- YAML dependency strategy.
- Unused or compatibility config fields.
- Any coordinate, IMU, RSSI, LAI, topology, calibration, or schema changes.
